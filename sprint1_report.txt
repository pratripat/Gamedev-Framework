=== Task 1: Universal Tween Framework ===

Added a complete tween system under scripts/utils/tween.py with:

- 31 easing functions (linear, quad, cubic, quart, quint, sine, expo, circ,
  elastic, back, bounce — each in in/out/in-out variants)
- Tween class supporting: property animation on any Python object, chaining,
  delays, callbacks (on_start, on_update, on_complete, on_cancel),
  cancellation, ping-pong mode, repeat count, infinite repeat
- TweenSystem class: lifecycle management (add, remove, cancel_all, update)
  with swap-with-last removal for O(1) removal
- Convenience methods: to(), from_to(), after() (for delayed callbacks)
- Automatic lerp for float, int, pygame.Vector2, pygame.Color, tuples/lists

This is an additive change. No existing code was refactored to use the tween
system. It is available for future systems (camera, UI, VFX, combat).

=== Task 2: Generic Object Pool ===

Added scripts/utils/object_pool.py and refactored two existing systems:

NEW:
- ObjectPool class: generic pre-allocating pool with factory pattern,
  free-list tracking, optional auto-growth (up to configurable max), usage
  statistics (capacity, active, free, utilization, peak, total acquired/
  released, grow count), reset() method, release_all() for batch release
- Compatible with the index-based access pattern: pool[idx] returns object

REFACTORED:
- FastProjectileSystem (scripts/systems/combat/fast_projectile_system.py):
  Replaced self.projectiles list + self.free_indices list with
  self.pool = ObjectPool(FastProjectile, ...). All internal and external
  access via self.pool[idx] instead of self.projectiles[idx].
  Pool grow=True, max_capacity=8000.

- ParticleEffectSystem (scripts/systems/rendering/particle_effect_system.py):
  Replaced self.particles list + self.free_indices list with
  self.pool = ObjectPool(FastParticle, ...). All access via self.pool[idx].
  Pool grow=True, max_capacity=8000.

EXTERNAL FILES UPDATED:
- scripts/systems/combat/destructible_system.py (projectiles -> pool)
- scripts/systems/rendering/render_system.py (projectiles -> pool)
- scripts/scenes/game_scene.py (projectiles -> pool, particles -> pool,
  respawn code uses pool.reset() instead of manual free list reset)

=== Task 3: Typed Event System ===

Added scripts/utils/events.py and updated EventManager:

NEW:
- GameEvent base class for all typed events
- 13 typed event dataclasses: DamageEvent, DeathEvent, ShootEvent,
  ProjectileCollisionEvent, WalkEvent, WaterSplashEvent, ScreenShakeEvent,
  DashStartEvent, AnimationFinishedEvent, CollisionEvent, SpawnGhostEvent,
  RemoveEntityEvent, BombBurstEvent
- TYPED_EVENT_MAP: bidirectional mapping between typed event classes and
  GameSceneEvents enum values
- All events carry typed fields instead of **kwargs

CHANGED (EventManager):
- Added _typed_subscribers dict for separate typed subscriber storage
- subscribe_typed(event_class, callback): registers for typed events
- emit_typed(event): dispatches to typed subscribers by event class
- Typed and legacy systems DO NOT cross-talk — prevents signature mismatches
- unsubscribe_all_for() also cleans typed subscribers
- Removed emit_walk() and emit_collision() special-case methods

UPDATED EMITTERS:
- physics_engine.py: emit_walk -> emit(GameSceneEvents.WALK, ...)
- physics_engine.py: emit_collision -> emit(GameSceneEvents.COLLISION, ...)

UPDATED SUBSCRIBERS:
- game_scene.py: _on_walk now accepts **kwargs instead of positional args

6. COMPATIBILITY CONCERNS
================================================================================

NONE. All changes are backward compatible:

- Tween framework: purely additive. No existing code depends on it.
- Object pool: FastProjectileSystem and ParticleEffectSystem maintain the
  same external API (active_indices, spawn/emit methods, collect_render_items).
  Any external code accessing pool[idx] works identically to old
  projectiles[idx]/particles[idx]. pool.release() internally sets active=False.
- Typed events: emit() with kwargs still works exactly as before.
  subscribe() with kwargs still works exactly as before.
  subscribe_typed() is additive. emit_typed() is additive.
- _on_walk signature change: was def _on_walk(self, pos, vel, entity_id),
  now def _on_walk(self, **kwargs). This is the only subscriber for WALK,
  so no external breakage.
- emit_walk/collision removal: only two call sites (both in physics_engine.py)
  were updated to use emit() in the same commit.

================================================================================
8. PERFORMANCE IMPACT
================================================================================

NEUTRAL to POSITIVE:

- ObjectPool: The pool internally uses the same free-list and pre-allocation
  pattern as the old ad-hoc implementations. The overhead of the pool wrapper
  is negligible (a few method calls per acquire/release). The pool's
  release() sets active=False internally, matching the old code pattern.
  The automatic growth (grow=True) prevents silent projectiles/particle loss
  at 4000 cap — pool can grow up to 8000 if needed.

- Tween Framework: Zero runtime overhead unless used. Tween objects are
  lightweight (no __slots__ but minimal attributes).

- Typed Events: emit_typed() does a dict lookup by event class (O(1)) plus
  iteration over typed subscribers. No performance regression for existing
  emit()-based code paths.

- The pool.release() method uses list.remove() for the active list cleanup.
  However, the hot path (projectile/particle update loop) uses the write_ptr
  compaction pattern and calls pool.release() for dying objects, which
  avoids list.remove() overhead by design.

================================================================================
9. TECHNICAL DEBT REMOVED
================================================================================

1. Duplicated free-list/active-list pattern in FastProjectileSystem and
   ParticleEffectSystem -> consolidated into ObjectPool

2. Special-case emit_walk() and emit_collision() methods -> replaced with
   standard emit(), removing non-obvious API surface

3. Positional argument mismatch in event subscribers -> _on_walk now uses
   consistent kwargs pattern matching all other event subscribers

4. Fixed capacity with silent failure -> pool now auto-grows up to 8000
   (configurable), preventing mysterious "projectiles stopped working" bugs

================================================================================
10. POTENTIAL RISKS
================================================================================

LOW:

1. Pool growth: If a bug rapidly allocates projectiles/particles, the pool
   can grow up to 8000. This is a safety net, not a risk — the old 4000
   hard cap was a silent failure point. The warning would be visible in
   stats if monitoring is added.

2. pool.release() calls list.remove() on the active list if the caller
   hasn't already removed it via write_ptr compaction. This only matters
   if some new code calls release() on projectiles still in active_indices.
   All existing call sites manage active_indices manually via the write_ptr
   pattern, so this is not an issue.

3. The TYPED_EVENT_MAP import in event_manager.py creates a dependency on
   scripts/utils/events.py. This file is small and stable. No circular
   import risk (events.py imports GameSceneEvents, not EventManager).
